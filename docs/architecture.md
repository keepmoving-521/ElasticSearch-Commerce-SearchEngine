# Architecture

## 核心原则

1. **领域优先**：商品、搜索、推荐、用户画像分别拥有自己的模型与规则。
2. **依赖倒置**：领域层声明端口，基础设施层实现 Elasticsearch、Redis、数据库等适配器。
3. **读写分离**：商品主数据以 PostgreSQL 为准，Elasticsearch 承担搜索读模型。
4. **事件驱动**：商品变更、用户行为通过 Kafka 异步传播，避免核心链路强耦合。
5. **可观测性优先**：统一结构化日志、请求 ID、指标和链路追踪。
6. **渐进式演进**：先保持模块化单体；仅在流量、团队边界或故障隔离要求明确时拆服务。

## 分层依赖

```text
API -> Application -> Domain
             ^
             |
       Infrastructure
```

- `domain`：实体、值对象、领域服务和仓储协议，不引用框架。
- `application`：搜索、推荐、商品同步等业务用例。
- `infrastructure`：数据库、缓存、搜索引擎和消息队列的具体实现。
- `api`：HTTP 路由、鉴权、校验和协议模型。

## 计划中的业务模块

| 模块 | 责任 |
| --- | --- |
| Catalog | 商品、类目、品牌、库存和价格的搜索读模型 |
| Search | 召回、过滤、排序、纠错、联想与聚合 |
| Recommendation | 候选召回、特征获取、排序和策略编排 |
| User | 用户画像、偏好和行为事件 |

## 演进路线

1. 建立商品模型和 Elasticsearch 索引生命周期。
2. 实现关键词搜索、过滤、分页和聚合。
3. 接入 Kafka 商品同步链路和幂等消费。
4. 建立行为采集、热门推荐和个性化召回。
5. 引入监控、压测、灰度发布及排序实验平台。
